# Запрос датасета с параметрами для TVF (первые два аргумента — параметры)

Если первые два аргумента функции `[dbo].[fnAZ_Rep_MainW]` должны приходить из фильтров дашборда (дата и список рейсов/групп), запрос в источнике датасета нужно писать с плейсхолдерами параметров.

## Формат подстановки параметров

В подзапросе (subsql) источника датасета DataLens подставляет значения параметров по плейсхолдерам:

```text
{{parameter_id}}
```

где **parameter_id** — это **id (guid)** параметра в датасете. Часто при создании параметра с именем «dte» его id тоже будет `dte`, тогда в запросе можно писать `{{dte}}`. Если id параметра — UUID, в запросе нужно использовать именно его: `{{uuid}}`. Один плейсхолдер — один параметр по его **guid**. Для работы превью и кнопки «Применить» в диалоге источника у параметров должны быть заданы **значения по умолчанию** — тогда они подставятся при проверке запроса.

## Какой запрос писать

1. В датасете создайте **два параметра** (например, для даты и для списка рейсов).
2. Запомните или скопируйте **id** каждого параметра (в интерфейсе датасета — в настройках поля параметра или в списке полей).
3. В источнике датасета выберите тип **«Подзапрос»** (MSSQL Subselect) и введите запрос в таком виде:

```sql
select * from [dbo].[fnAZ_Rep_MainW](
  {{<id_параметра_даты>}},
  {{<id_параметра_рейсов_или_группы>}},
  null,
  null
)
```

Подставьте вместо `<id_параметра_даты>` и `<id_параметра_рейсов_или_группы>` реальные guid параметров (без угловых скобок).

**Пример** (если id параметров, например, `a1b2c3d4-...` и `e5f6g7h8-...`):

```sql
select * from [dbo].[fnAZ_Rep_MainW](
  {{a1b2c3d4-0000-4000-8000-000000000001}},
  {{e5f6g7h8-0000-4000-8000-000000000002}},
  null,
  null
)
```

## Формат значений

В YDL OS значения параметров в подзапросе подставляются как **SQL-литералы в одинарных кавычках** (дата и строка — всегда в кавычках, запятые внутри строки не разбивают аргументы).

- **Первый аргумент (дата):** подставляется в кавычках. Если в превью «нет данных» при формате `01/09/2024`, задайте значение по умолчанию и формат фильтра как **YYYY-MM-DD** (например `2024-09-01`) — так данные часто возвращаются корректно.
- **Второй аргумент (рейсы/группы):** одна строка в кавычках, например `'B2731,B2732'`. Значение по умолчанию можно задавать без пробелов (`B2731,B2732`) или с пробелами — в запрос уйдёт одна строка в одинарных кавычках.

**Дашборд:** чтобы по кнопке «Применить» в запрос подставлялись выбранные в фильтрах дата и рейсы, фильтры должны быть **привязаны к параметрам датасета** (dte и nrs датасета Repka). Без привязки бэкенд подставляет только значения по умолчанию.

**Варианты фильтров только из справочников:** для «Группа» и «Рейсы» в настройках виджета укажите **источник списка «Из датасета»** — датасеты на таблицах AZ_FLIGHTS_GROUPS и AZ_FLIGHTSinGROUPS и нужные поля. Тогда пользователь выбирает только значения из этих таблиц; выбранные коды передаются в параметр nrs (и при необходимости в параметр группы).

## Включение подстановки в подключении

Чтобы плейсхолдеры `{{...}}` в подзапросе обрабатывались, в настройках **подключения** MSSQL должна быть включена опция, разрешающая использование шаблонов в источниках (например, «Разрешить подстановку параметров в запросах» / allow template). Без этого подзапрос уйдёт в БД без подстановки.

## Превью и проверка источника в диалоге

При сохранении/проверке источника в диалоге «Источник» запрос может уходить в БД **без** переданных значений параметров. Тогда плейсхолдеры не подставляются, в запросе остаются `{{...}}`, и MSSQL выдаёт ошибку (например, «The batch could not be analyzed because of compile errors»). Чтобы превью и кнопка «Применить» в диалоге источника работали, у параметров датасета стоит задать **значения по умолчанию** (например, дата `01/09/2024` и строка `B2981, B2941`). Тогда при проверке источника подставятся они.

## Дата и календарь-диапазон (по месяцу)

Для даты **id всё равно нужен**: в датасете создаётся **один параметр** (например, «Дата» или `dt`). У него появляется свой id (guid). В запросе в подзапросе вы подставляете этот id в плейсхолдер: `{{id_параметра_даты}}`.

На дашборде вы не «придумываете» id для календаря — вы вешаете фильтр **типа «Календарь» / «Диапазон дат»** (по месяцу) и в настройках этого элемента указываете **источник: параметр датасета** — тот самый параметр даты, который создали в датасете. То есть:

1. В датасете: создать параметр для даты (название любое, например «Дата»). Скопировать его **id** и вставить в запрос: `{{этот_id}}`.
2. На дашборде: добавить элемент управления «Календарь» / «Диапазон дат» (по месяцу), в источнике выбрать **этот параметр** датасета (по имени параметра).

Так «календарь-диапазон по месяцу» просто задаёт **значение** параметра даты; сам параметр и его id живут в датасете.

Если функция ожидает один параметр даты (например, первый день месяца), значение из фильтра «по месяцу» обычно нужно приводить к одному дню на бэкенде или в настройках (например, «начало периода»). Это уже отдельная настройка логики даты.

## Связь с дашбордом

- На дашборде создайте элементы управления: **календарь-диапазон (по месяцу)** → привязать к **параметру даты** датасета; фильтр по рейсам/группе → привязать к **параметру рейсов/группы**.
- При нажатии «Применить» выбранные значения уйдут в запрос как `parameter_values` и подставятся вместо `{{parameter_id}}` в подзапрос.

Итог: в датасете два параметра (дата и рейсы/группа), у каждого есть id. В запросе — `{{id_даты}}` и `{{id_рейсов}}`. На дашборде календарь по месяцу привязывается к параметру даты по имени, без отдельного «id для календаря».

## Ошибка «Default value for field … is not valid» (PARAMETER_VALUE_INVALID)

Если при сохранении датасета или при установке значения по умолчанию появлялась ошибка `ERR.DS_API.SOURCE_CONFIG.PARAMETER_VALUE_INVALID` для значений вроде `01/09/2024` или `B2731, B2732`, причина была в том, что в бэкенде (datalens-backend) ограничение по умолчанию для строковых параметров допускало только символы `[a-zA-Z0-9]*`. В форке YDL OS это исправлено: (1) класс `DefaultParameterValueConstraint` использует pattern `.*`; (2) для уже сохранённых датасетов с ограничением типа regex и pattern `[a-zA-Z0-9]*` любая строка считается допустимой (обратная совместимость). Код: `components/datalens-backend/lib/dl_core/dl_core/fields.py`.

Используйте образы aeronavigatorbi (`.\run-own-images.ps1`); код backend уже в образе. После изменения кода пересоберите образы (`.\build-all-images.ps1`) и перезапустите или выполните `docker compose -f docker-compose.yaml -f docker-compose.own-images.yaml restart control-api data-api` из каталога `datalens/`.

**Если ошибка не пропадает:** (1) Запускайте стек через `.\run-own-images.ps1` из каталога `datalens/`. (2) Проверьте образ: `docker exec datalens-control-api-prod grep -n "Backward compatibility" /venv/lib/python3.10/site-packages/dl_core/dl_core/fields.py`. (3) После правок в backend пересоберите образы и перезапустите.
