# Устранение неполадок: иконки, подключения MSSQL, ошибка датасета

## Режим «всё в исходниках» (без патчей к бандлу)

- Сборка UI: `.\build-and-run-ui.ps1` (из папки datalens) — собирает components/datalens-ui, образ aeronavigatorbi/datalens-ui, перезапуск ui/ui-api. Патчи к dl-main.*.js не применяются.
- Проверка, что крутится наша сборка: открыть http://localhost:8080/build/build-info.txt — должно быть «Aeronavigator BI custom build».
- Иконки и брендинг — в исходниках (opensource-branding.scss, icons, layout). После правок — пересборка и перезапуск через build-and-run-ui.ps1.

## Превью сводной таблицы: только спиннер

Если в визарде сводной справа вечно крутится загрузка и данные не появляются:
- **Network (DevTools):** есть ли запрос за данными графика к data-api/control-api, какой статус (200 / 5xx / таймаут).
- **Console:** ошибки JS или сообщения с бэкенда.
- **Бэкенд:** контейнеры data-api, control-api запущены; для сводной на MSSQL нужны правки в data-api (см. backend-patches, docs/wizard-pivot-mssql-error.md).
Подробнее: docs/PATCHES-AND-UI-FROM-IMAGE.md, раздел «Превью сводной таблицы».

## Иконки слева (сайдбар)

- Иконки задаются в исходниках (навигация, layout). При пустых иконках проверьте, что используется образ с нашей сборкой (build-info.txt) и что в сборку входит opensource-branding.scss и конфиг иконок.

## Подключения MSSQL не отображаются

- В `docker-compose.yaml` для **control-api** задано:
  `CONNECTOR_AVAILABILITY_VISIBLE: clickhouse,postgres,chyt,ydb,mysql,greenplum,mssql,oracle,trino,appmetrica_api,metrika_api`
- Список коннекторов отдаёт control-api (`/api/v1/info/connectors`). Если MSSQL нет в списке:
  1. Проверьте, что контейнер **control-api** запущен и без ошибок: `docker logs datalens-control-api-dev --tail 50`.
  2. Убедитесь, что при запуске в контейнер действительно попадает переменная `CONNECTOR_AVAILABILITY_VISIBLE` (образ может читать её из окружения). При необходимости перезапустите: `docker compose -f docker-compose.yaml restart control-api`.
  3. Если используете свой backend из репозитория (не образ), в `app/app_configs/api.yaml` в блоке `CONNECTOR_AVAILABILITY` должен быть `mssql` в `uncategorized`; при пустом `visible_connectors: []` образ может подставлять видимые коннекторы из env.

## Ошибка «Некорректный запрос к датасету»

- Обычно связана с источником датасета (запрос, параметры, типы полей) или с data-api.
- Проверьте:
  1. Логи **data-api**: `docker logs datalens-data-api-dev --tail 100`.
  2. Код backend в образах aeronavigatorbi. После изменений: пересборка `.\build-all-images.ps1` и перезапуск или `docker compose -f docker-compose.yaml -f docker-compose.own-images.yaml restart data-api`.
  3. Параметры датасета и источник (подзапрос/TVF) — см. `docs/dataset-tvf-parameters.md`, `docs/wizard-pivot-mssql-error.md`.

## Зелёная галочка после «Проверить подключение»

- В коде UI откачены все правки (см. docs/REVERTED-UI-FIXES-AND-ROOT-CAUSE.md): логика проверки подключения и отображения иконки снова как в оригинальном DataLens. Если галочка не появляется при 200 от verifyConnection, возможные причины: SDK бросает исключение при разборе ответа; или используется другая сборка/образ UI (вариант B vs дефолтный). Проверьте в Network ответ verifyConnection и в Console — ошибки после нажатия «Проверить подключение».

## Палитры (Настройки сервиса → Палитры): не отображаются превью, 403

- В коде UI откачены правки (см. docs/REVERTED-UI-FIXES-AND-ROOT-CAUSE.md): при ошибке загрузки палитр fallback не подставляется; тост при ошибке смены палитры по умолчанию снова с полным error (кнопка «Подробнее»). Если превью нет — при 403 или другой ошибке `getColorPalettesList()` список не приходит и остаётся пустым; причина на стороне бэкенда/прав или способа раздачи UI (вариант B).
- Превью рисуются компонентом `AutogeneratedPaletteIcon` (CSS gradient). Если превью не видно: убедитесь, что API `getColorPalettesList()` (us) возвращает данные; при ошибке или пустом ответе список пустой. Проверьте в Network/Console запросы к палитрам и ошибки 403.
