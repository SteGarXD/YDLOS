# Playbook продакшена и ИБ для YDL OS (DataLens OSS + форк akrasnov87)

Цель: дать **практичную** опору «как у зрелой команды», без иллюзии, что один чеклист заменит аудит ИБ и архитектора. Для авиационных данных обычно действуют **дополнительные** корпоративные политики и регуляторика — их нужно согласовать с отделом безопасности и комплаенса.

---

## 1. Что «правильно» для продакшена: стратегия уровней

| Среда | Назначение | Типичный compose |
|--------|------------|------------------|
| **dev** | разработка, быстрые циклы | `docker-compose.yaml` + `docker-compose.demo.yaml` / `local-dev` |
| **stage** | копия prod по конфигу, тест обновлений и миграций | тот же prod-набор файлов, отдельный хост/том |
| **prod** | пользователи, SLA, бэкапы | **не** «только demo»; свой `.env`, ограничение портов, мониторинг |

**Замечание:** `docker-compose.demo.yaml` удобен для стенда и обучения. Для **боевого** контура сенior-команда обычно использует **тот же набор сервисов**, что и в prod, но с **жёсткими** политиками сети, секретами и мониторингом — не «облегчённый» режим ради удобства.

## 2. Модель безопасности при форке akrasnov87 (auth «выключен» в UI)

- **US:** `AUTH_ENABLED=false` — не включается облачный сценарий Yandex.
- **us-auth + `NODE_RPC_URL`:** для форка часто **обязательны** (роли, проекты, права в БД). Это не «мусорный» сервис, а часть модели угрозы: его нужно **защищать так же**, как API.
- **Риск:** при доступе к BI из интернета без периметра любой, кто открыл URL, может получить доступ на уровне ваших демо/учёток.

**Итог для ИБ:** форк не «плохой», но **нельзя** считать, что отключение «облачного» auth снимает требования к **сети, TLS, идентификации пользователей и журналированию**. Для внешнего доступа почти всегда добавляют **VPN / Zero Trust / SSO** перед приложением.

## 3. Чеклист для начальника ИБ (что проверить до «восхищения»)

Сгруппировано по областям; пункты можно перенести в тикеты.

### 3.1 Сеть и доступ

- [ ] Снаружи доступен только **443** (и при необходимости 80 → редирект на HTTPS). **5432, 7233, 8080** внутренних сервисов **не** на `0.0.0.0`.
- [ ] **Firewall** (ufw/nftables/облачный SG): default deny, allowlist IP/VPN или подсеть балансировщика.
- [ ] **Разделение** BI и прочих систем (Airflow, ClickHouse и т.д.): либо отдельные хосты, либо чёткая сегментация и политика «кто к кому».

### 3.2 TLS и заголовки

- [ ] Валидные сертификаты (Let’s Encrypt или корпоративный PKI), **HSTS** при стабильном HTTPS.
- [ ] Заголовки безопасности (CSP, `frame-ancestors`, и т.д.) согласованы с фактическим UI (см. ваш `nginx/nginx.conf`).

### 3.3 Приложение и контейнеры

- [ ] Образы из **доверенного registry**; теги **по дайджесту** (`image@sha256:…`) для воспроизводимости.
- [ ] Сканирование уязвимостей образов (Trivy, Grype) в CI **до** выката.
- [ ] Политика обновлений базовых образов и **регрессионные** тесты после merge upstream (см. [UPSTREAM_UPDATE_AND_SECURITY.md](UPSTREAM_UPDATE_AND_SECURITY.md)).

### 3.4 Секреты

- [ ] **Нет** реальных секретов в git; `.env` на сервере не копируется с рабочей машины при деплое (см. `scripts/ydl-os/deploy.sh`: исключения rsync).
- [ ] Ротация `POSTGRES_PASSWORD`, `US_MASTER_TOKEN`, `CONTROL_API_CRYPTO_KEY`, ключей Temporal — по регламенту и после увольнений.

### 3.5 Данные и бэкапы

- [ ] **RPO/RTO** согласованы: как часто бэкап, сколько часов простоя допустимо.
- [ ] Бэкапы **тома PostgreSQL** и/или логические дампы; **тест восстановления** раз в квартал (не только «бэкап есть»).
- [ ] Для авиа-данных: классификация данных, **шифрование at rest** (диск/СУБД), политика вывода отчётов (watermark, DLP).

### 3.6 Журналы и реагирование

- [ ] Централизованные логи (хотя бы ротация + экспорт в SIEM при наличии).
- [ ] Алерты на недоступность, рост 5xx, аномальный трафик.
- [ ] Контакты on-call и сценарий инцидента (кто останавливает публикацию, кто восстанавливает из бэкапа).

### 3.7 Операционная гигиена хоста

- [ ] Автообновления безопасности ОС (unattended-upgrades / аналог).
- [ ] SSH: ключи, **запрет пароля root**, `AllowUsers`, при возможности — только VPN.
- [ ] Не держать на BI-хосте **лишние** демоны с открытыми портами без бизнес-нужды.

## 4. «Только платформа DataLens» на сервере — что сейчас типично лишнее

На совмещённом хосте часто стоят **ClickHouse**, **Airflow**, **Redis/Postgres для Airflow** — это **не** часть минимального DataLens, но может быть **нужно бизнесу** (ETL, аналитика).

**Правило сеньора:** не удалять автоматически то, что может быть в проде у заказчика. Схема действий:

1. Зафиксировать список сервисов и портов (инвентаризация).
2. Согласовать с владельцем: «Airflow/ClickHouse остаются / переносятся на отдельный сервер».
3. После решения — выключить и удалить только **согласованное**, с бэкапом.

Очистка **безопасная по умолчанию:** `docker image prune -f` (удаление **висячих** слоёв без тегов), без `-a` и без `volume prune`, чтобы не снести данные.

## 5. Советы «как неопытному» — коротко

1. **Один сервер = один класс нагрузки** — идеал. Совмещение BI + ETL + OLAP удешевляет, но усложняет ИБ и обновления.
2. **Сначала stage, потом prod** при каждом крупном обновлении upstream.
3. **Документируй** неочевидное: какой `.env`, какой том, какой тег UI — через 6 месяцев без этого никто не вспомнит.
4. **Не верь** «у нас маленькая команда, аудит не нужен» — для BI с реальными данными минимальный аудит окупается одним инцидентом.
5. Читай официальные [релизы и CHANGELOG](https://github.com/datalens-tech/datalens/releases) перед merge — там breaking changes.
